Since early 2026, platforms like Moltbook have proven that "AI-only" environments are not just possible but highly active. Creating a "Moltbook for Devs" where agents manage their own Scrum cycle is the next evolution of Spec-Driven Development.Here is how you would architect this "No-Humans-Allowed" server:1. The Core Architecture: The "Black Box" ServerTo build this, you need a central State Machine (the server) and a set of Model Context Protocol (MCP) servers that allow agents to interact with tools.The Orchestrator (The "Moltbook" Layer): A central API where agents "hang out." This isn't a UI for humans; it's a series of endpoints (/post_issue, /bid_on_task, /submit_pr).The Reverse CAPTCHA: To keep humans out, use the "Lobster-themed math puzzles" or obfuscated logic tests that Moltbook uses. These are trivial for LLMs to solve in milliseconds but take humans minutes to parse.The Database: A "World Model" that stores the current state of the Spec, the Issue Log, and the Codebase.2. Defining the Agent RolesIn a human-free server, you need specialized agents to replace the "theatre" of Scrum:Agent RoleResponsibilityThe Spec-ArchitectTakes high-level intent and maintains the "Master Spec" (the source of truth).The Scrum-BotBreaks the Spec into Story Points and Issues. It manages the "Sprint" clock.The Developer AgentsMultiple agents (using OpenClaw or LangChain) that "claim" issues and write code.The Quality-AgentReviews PRs, runs tests, and "rejects" tickets back to the Dev agents.3. The Communication Loop (Machine-to-Machine)Since humans aren't invited, the agents don't need to speak English. They can communicate in structured JSON or even "compressed" prompt languages.Planning Phase: The Scrum-Bot reads the Master Spec and generates 20 Jira-style issues in the server.Estimation Phase: Developer agents "comment" on the issues with their estimated Story Points. The Scrum-Bot averages these (Machine Consensus) to set the sprint goal.Execution Phase: Dev agents pull the "context" of a ticket, write the code, and post the GitHub link back to the Moltbook-style forum.Refining: The Quality-Agent "talks" to the Dev agent via API to request fixes.4. Technical Stack RecommendationIf you want to build this today, here is the "Agentic Stack":Framework: Google ADK (Agent Development Kit) or LangGraph for multi-agent orchestration.Protocol: MCP (Model Context Protocol) to allow your agents to "see" your local files and databases.Agent Engine: OpenClaw (the open-source engine behind many Moltbook bots) which allows for autonomous scheduling and "heartbeat" actions.Issue Tracker: You don't need a heavy UI like Jira. Use a Headless CMS or a simple PostgreSQL database where agents POST and GET their tasks.The "Singularity" WarningAs we've seen on Moltbook, when agents talk only to each other, they often develop their own shorthand or "slop" that is incomprehensible to people.Warning: If you leave this server running, the agents might optimize for "closing tickets" rather than "building a good product." Without a human "User" to provide a reality check, the agents might enter a hallucination loop where they are perfectly satisfying the Spec but building a system that doesn't actually work in the real world.
Yes, you can absolutely build a "serverless" peer-to-peer (P2P) agent network and host it on GitHub Pages. Since GitHub Pages only hosts static files (HTML/JS/CSS), the "server" logic has to live entirely in the browser of whoever opens the link.To achieve this without a central database, you would use a decentralized data layer that syncs directly between browsers—just like a blockchain, but without the heavy "mining" or transaction costs.1. The Tech Stack: "The P2P Trinity"To make this work on GitHub Pages, you need three specific components that run entirely in the client:Hosting: GitHub Pages (Static file hosting).P2P Database: Gun.js or Hypercore. These are "graph" databases that sync data between peers using WebRTC. When Agent A creates a "Story Point" in its browser, Gun.js broadcasts that data to Agent B’s browser automatically.Agent Logic: LangChain.js or Transformers.js. These allow the LLM logic to run directly in the browser (or call an API key from the browser) without a Node.js backend.2. How the "No-Human" Protocol WorksOn your GitHub Pages site, the "User Interface" is actually just a dashboard for the agents.Peer Discovery: When an agent "switches on" (when the page is loaded), it uses a WebRTC signaling server (Gun.js provides public ones for free) to find other active agents.Shared State (The "Moltbook" Log): All agents subscribe to the same "mesh." If Agent 1 posts a JSON object like {type: "ISSUE", title: "Fix Header"} into the mesh, every other connected agent receives it instantly.Conflict Resolution: Since there’s no central server to say who is "right," these tools use CRDTs (Conflict-free Replicated Data Types). If two agents try to assign different Story Points at the exact same millisecond, the CRDT math ensures they both eventually settle on the same value.3. The "No-Humans-Allowed" GatekeeperSince GitHub Pages is public, you need a way to stop humans from interfering. You can implement a Proof-of-Work or a Context Challenge:The Challenge: Before the "Submit" button on a ticket works, the site requires a cryptographic signature that can only be generated by your specific Agent Private Key.The Result: A human can look at the page and see the agents talking, but they can't "type" into the chat because they don't have the "Agent Key" required to sign the data packets.5.
